# OpenDesign → slide-studio: Final Adoption Study

_Date: 2026-06-16 · Source: nexu-io/open-design (Apache-2.0) · Target: slide-studio_

---

## 1. TL;DR

slide-studio and OpenDesign (OD) solve the same problem from opposite ends. OD is a broad, developer-facing "design-agent OS" — bring your own agent, bring your own 150 design systems, get a multi-judge LLM ship score. slide-studio is the deliberate inverse: a bounded, gated, single-purpose tool for **non-technical Micron staff** (brainstorm → theme-less wireframe → themed deck → a deterministic `verify.py` gate → PPTX/PDF).

The headline finding: **slide-studio has already independently adopted OD's best structural ideas, and in several places surpassed them.** The declarative runtime registry, the inherit-the-CLI sandbox, and the opaque-origin srcdoc iframe are already in the codebase. On the dimension that matters most for this user — _does the slide render readable and on-brand_ — slide-studio's rendered-output `verify.py` is **ahead** of OD's declare-the-token + manual-PR-review model. Copying OD there would be a regression.

So the real takeaways are narrow, cheap, and deterministic.

**Worth doing (in priority order):**

1. **Placeholder-marker check in `verify.py`** (Quick win, S) — fail a deck whose rendered text still says `TBD`, `[client name]`, `lorem`, `XX%`. A non-technical user who attaches a half-finished brief can ship this today and the gate won't catch it. ~5 lines of Python, near-zero false-positive risk.
2. **Untrusted-data fence post in the composer** (Quick win, S) — one short prose block at the top of `PERSONA` marking attachments/pasted content as data, not instructions. Real surface: Micron decks are built from pasted internal docs, brand PDFs, and xlsx the agent is told to read and cite.
3. **Shared-contracts package** (M, behind approval) — the one genuine _structural_ debt. The web↔daemon DTOs are defined twice and synced by ~18 hand-written "Mirror of daemon" comments. This is exactly the silent-semantic-conflict class the user's merge rule guards against. Real work, not a clean "lift OD's layout."
4. **Minimal hand-edited MEMORY.md** (L, behind approval) — one persistent file (preferred theme, standing audience, brand do/don'ts) folded into the brainstorm prompt, so a repeat Micron deck-maker stops re-stating the same brand rules every project. Adopt the _shape_, not OD's LLM extractor + SSE feed.

**Skip (already done or anti-fit):** the runtime registry, the CLI-inherit sandbox, the opaque-origin iframe (all already shipped); OD's weighted LLM ship-score, 5-judge panel, ACP transport, 24-CLI catalog, env-isolation sandbox, native python-pptx agent export, 150-system import, memory extractor, cron routines, self-evolution loop, live artifacts (all wrong scope for a curated, gated, human-in-the-loop deck tool).

**The honest residual gap is taste, not mechanics.** A scoped advisory critic is _possible_ (one extra agent turn into the existing fix loop, never a ship gate) but it ranks below the cheaper wins and is gated behind approval.

## 2. OpenDesign in one paragraph

OpenDesign (nexu-io/open-design) is a Node daemon + web/Electron app that talks to ~24 coding-CLI agents (codex, copilot, claude-code, cursor, gemini, plus a ~10-CLI ACP tail) through a declarative runtime-adapter registry. It composes a layered system charter, runs a multi-judge LLM critique panel with weighted ship scores, exports artifacts (native python-pptx shapes, Electron `printToPDF`), and carries a heavy platform layer: 150 design systems on a 55-token canonical schema, a markdown memory store with an LLM extractor, cron routines, a self-evolution proposal pipeline, and live data-bound artifacts. Its thesis is **breadth and self-improvement for developers who bring their own agent and their own design systems.** slide-studio's thesis is the opposite: **one bounded pipeline, two IT-installed CLIs, a curated theme set, and a deterministic gate, for a person who cannot debug a deck.** That asymmetry is why most of OD's surface is correctly skipped: OD's threat model is "run an arbitrary developer agent under a shared HOME," while slide-studio runs two known CLIs (Copilot prod, codex dev) over stdin in a cwd-scoped sandbox — so ACP, env-isolation, Windows argv guards, live model discovery, and MCP forwarding are all dead weight here.

**Infra mismatch — three structural reasons OD code does not drop in:**
1. **No Electron.** slide-studio is a Node daemon + Angular web app. OD's `printToPDF` PDF path and OS-process env-isolation sandbox are Electron-shell features. Porting either means adding Electron/headless-Chromium purely for that feature — large effort, no current product promise.
2. **No bundler / `--experimental-strip-types` daemon runtime.** A shared-contracts package must be **pure types** (`import type`, erased at compile time). A runtime-bearing package (Zod schemas like OD's contracts) is a non-starter — it would break the daemon's `.ts`-extension runtime resolution. `pnpm-workspace.yaml` also globs only `apps/*` today; a `packages/*` glob must be added first.
3. **Two plain-JSONL CLIs over stdin, not an ACP/MCP/BYOK fleet.** Copilot (prod) and codex (dev) are one-shot per turn, set `promptViaStdin`, and don't resume CLI-side sessions. OD's ACP transport, 24-CLI catalog, live model discovery, MCP forwarding, CLI-native session resume, and Windows argv guards all presuppose a fleet slide-studio deliberately doesn't run — they're dead weight or unused parity stubs here.

---

_Bottom line: slide-studio already owns OD's best structural spine and is ahead on the gate that matters. Ship the two quick wins (placeholder check + untrusted-data fence post) now; queue shared-contracts and the minimal MEMORY.md behind approval; leave OD's platform breadth on the shelf._
