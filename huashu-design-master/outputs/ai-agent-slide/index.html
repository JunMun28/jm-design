AI Systems Primer
01 / Definition

An AI agent is a system that pursues a goal.

It uses a model to decide what to do next, calls tools, observes results, and repeats until the task is done or a boundary stops it.

Chatbot

Responds to a message. The user drives each step.

Agent

Works toward an outcome. The system drives the loop.

The agent loop

Useful agents combine reasoning, action, feedback, memory, and control.

Model decides next action Plans, selects tools, evaluates progress

Goal

What outcome should be achieved?

Plan

Break the task into next useful moves.

Tools

Search, code, browse, call APIs, write files.

Feedback

Read results, errors, approvals, and state.

Memory

Carry context across steps or sessions.

Guardrails

Permissions, policies, budgets, and human review.

Simple test

If it can choose and execute the next step toward a goal, it is agentic.

Important

Autonomy raises stakes: agents need visibility, limits, and rollback paths.

03 / Runtime loop

The runtime pattern is simple. The details are not.

Most useful agents repeat the same loop, but quality depends on the handoffs between steps.

01 / Understand

Convert the request into a working goal.

Clarify scope, missing inputs, success criteria, and constraints before action.

Output: task state
02 / Choose

Select the next action, not the whole fantasy plan.

Good agents make progress through small, verifiable moves.

Output: next step
03 / Use tools

Read, search, code, call APIs, or operate software.

The model is the planner; tools are how the system touches the world.

Output: external result
04 / Observe

Inspect the result instead of assuming it worked.

Errors, screenshots, tests, logs, and user feedback become the next context.

Output: evidence
05 / Decide

Continue, ask, stop, or hand off.

The strongest agents know when autonomy is no longer appropriate.

Output: control decision
06 / Record

Keep useful state, discard noise.

Memory should improve the next step, not accumulate every token forever.

Output: updated context

04 / What is inside?

An agent is a product architecture, not just a model call.

The model matters, but the surrounding system determines whether it is useful, observable, and safe.

01

Model

Reasoning, language, planning, tool selection.

02

Tools

APIs, browser, code execution, databases, files, apps.

03

State

Task memory, user context, intermediate artifacts, logs.

04

Control plane

Permissions, policies, budgets, approvals, rollback.

05

Interface

How people assign work, inspect progress, and intervene.

Agent quality is mostly system quality: tools, state, evals, and recovery paths.

A stronger model can still fail inside a weak product shell. A weaker model can be useful when the loop is constrained and observable.

05 / Where agents fit

Best fit: work that is goal-based, tool-heavy, and verifiable.

Agents are strongest when the path is uncertain but progress can be checked.

Research

Gather sources, compare claims, synthesize notes, keep citations visible.

High fit: many sources

Software work

Read code, patch files, run tests, inspect failures, iterate toward green.

High fit: testable output

Operations

Triage tickets, update systems, draft replies, escalate uncertain cases.

Medium fit: approvals needed

Data analysis

Query, clean, chart, explain anomalies, produce repeatable summaries.

High fit: audit trail

Purchasing or money movement

Can assist, but final action should require clear human confirmation.

Careful fit: high stakes

Creative production

Generate options, critique, revise, export assets, keep direction consistent.

Good fit: reviewable drafts

06 / Build responsibly

The key design question: how much autonomy should this system have?

Use agents where they reduce coordination cost. Add limits where mistakes are expensive.

1

Define the boundary

What can the agent do alone, what requires approval, and what is forbidden?

2

Make progress visible

Expose plan, tool calls, assumptions, sources, and current state.

3

Verify before completion

Use tests, screenshots, citations, logs, or human review before claiming done.

4

Design rollback

Every external action needs an undo path, audit trail, or containment strategy.

Operating rule

Autonomy is earned, not assumed.

Start with narrow permissions, strong observability, and clear handoff points. Expand only after the loop is reliable.

Practical takeaway: agent design is workflow design plus safety design.
