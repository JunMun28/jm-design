# Intake Questions

Use these as starting templates. Adjust wording to match the user's tone and what they've already revealed in the opening message.

Ask **one question at a time** when possible. If the user is moving fast and has clearly answered several in their opener, batch the remaining ones — but never ask more than ~5 total before the first draft.

Prefer **multiple-choice options the user can click** over open-ended questions, especially for audience, purpose, and length. Open-ended works better for scope.

## Question 1 — Audience

**Wording:**
> Who's the audience? Pick the closest fit (or tell me if it's something else).

**Options:**
- Internal team / colleagues at your company
- Executives / leadership / non-technical managers
- Customers / prospects
- Developers / engineers / technical staff
- External event audience (conference, meetup)
- Investors / board
- Students / training participants

Follow up if needed: "Roughly how technical are they on this topic — beginner, working knowledge, or expert?"

## Question 2 — Purpose / Goal

**Wording:**
> What does success look like after they see this deck?

**Options:**
- **Awareness** — they leave knowing this thing exists
- **Adoption** — they leave wanting to try it
- **Training / onboarding** — they leave able to do something new
- **Decision / approval** — they leave aligned on a yes/no
- **Pitch** — they leave wanting to fund / hire / partner with us
- **Status update** — they leave informed about where things stand

This often clarifies what the closing slide should look like (call-to-action vs. summary vs. ask).

## Question 3 — Scope

**Wording:**
> In one sentence, what's actually being covered?

This is the one open-ended question. The user's answer here becomes the deck's working subject line. Push back if it's too broad: "That's a 60-slide deck — what's the slice we cover today?"

If they're stuck, offer common slices: "definition + how it works + use cases + how to start" is a reliable training-deck spine.

## Question 4 — Length

**Wording:**
> Roughly how long should it be?

**Options:**
- **Short** — 3–5 slides · ~5 min talk
- **Medium** — 6–10 slides · ~15 min talk
- **Long** — 12–20 slides · 30+ min talk
- **Workshop** — 20+ slides with breaks / exercises

If a live demo is involved, factor that in: "Want me to leave a slide as the 'we open VS Code now' transition?"

## Question 5 — Reference style (optional)

**Wording:**
> Want me to match an existing visual style? If so, paste a path to a file (HTML, PDF, PPTX, or even a screenshot) and I'll extract the design grammar before drafting.

This is optional. If the user says no, you'll set design defaults in the DESIGN SYSTEM block based on what fits the audience and goal.

If yes:

1. Read the file with the Read tool.
2. Extract palette, typography, layout patterns, and any theme-specific overrides.
3. Restate them in the SHARED GRAMMAR and DESIGN SYSTEM sections.
4. Use real class names / token names from the source file.

## When to skip questions

Skip any question the user has already answered in their opener. Examples:

- "I want a 5-slide deck about X for my CEO" → audience and length answered, just ask about goal and scope.
- "Make a training deck for managers about Y" → goal and audience answered, ask length and scope.
- "Build me a quick brainstorm for Z" → no signal, ask all 4.

## After intake

Recap their answers in 3–5 lines before drafting. This gives them one last chance to correct before you commit to a direction.

```
Quick recap:
  Audience  – <answer>
  Goal      – <answer>
  Scope     – <answer>
  Length    – <answer>
  Style     – <answer or "I'll choose">

Drafting the brainstorm now. One moment.
```

Then write the file.
